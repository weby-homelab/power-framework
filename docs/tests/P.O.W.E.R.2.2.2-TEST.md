# 📊 Звіт про тестування, швидкодію та якість пошуку у P.O.W.E.R. v2.2.2

Цей звіт фіксує **фіналізацію** комплексного IR-дослідження фреймворку **P.O.W.E.R. v2.2.2**:
коректний бенчмарк усіх **5 режимів пошуку** на **повному реальному vault** (`/root/gemma/brain`, 565 `.md`-файлів) з **дуальним ground-truth** (лексичний + семантичний), що усуває методологічні суперечності попередніх звітів; аналіз **7 системних патернів відмов**, виявлених у історичних звітах (2.0.3 → 2.2.1); та інцидент з утилітою порівняння ембедінгів (витік 42 ГБ RAM).

Тестування проведено на сервері `WS` (Workstation, 126 ГБ RAM, 20 cores) станом на **18 липня 2026 року**.

---

## 🚨 1. Інцидент: витік 42 ГБ RAM (утиліта порівняння ембедінгів)

Під час дослідження крос-лінгвальної якості ембедінгів (Qwen3-Embedding vs MiniLM) запущена утиліта синхронізації під-вault у venv з `qwen3-embed`:

```
PID 90317  /tmp/power_test_venv/bin/python3 /tmp/power_test_venv/bin/power sync /tmp/sub_vault
%MEM 34.3%  ≈ 42 ГБ RSS
```

- **Симптом**: `free -m` показував `used=51486 MB` (базове навантаження WS — лише ~8 ГБ). Процес утримував Qwen3 ONNX-модель (1024d) + ONNXRuntime arena, яка не звільнялася.
- **Корінь**: спроба `pkill -f "qwen3_sub"` **не спрацювала** — процес запущений як `/tmp/power_test_venv/bin/power sync /tmp/sub_vault` (без підрядка `qwen3_sub` у команді). Спроба повного vault-embedding (1024d ONNX на CPU) також показала лише **8 документів за 20+ хв** — непридатно для повного vault.
- **Рішення**: прямий `kill -9 90317`. Після завершення `used` впав до **8032 МБ** (звільнено 42 ГБ).
- **Урок**:
    1. Для зупинки довгих процесів завжди фіксувати точний PID (`ps aux --sort=-%mem`) замість покладання на `pkill -f` з неточним патерном.
    2. Qwen3-Embedding (1024d) на CPU **непридатний** для повного vault (565 файлів) — latency синхронізації > година. Крос-лінгвальне порівняння моделей слід робити **ізольованим cosine-тестом** на парах речень (UA↔EN), а не повним vault-embedding.
    3. Додати у `power sync` м'який ліміт RSS / попередження при >10 ГБ утримання, щоб уникнути тихого роздування на багатоядерних хостах.

## 📚 2. Історичний аналіз: 7 системних патернів відмов (2.0.3 → 2.2.1)

Прочитано всі 12 попередніх TEST-звітів. Виявлено **7 повторюваних патернів**, що пояснюють розбіжності між звітами:

### FP-1 — Hybrid RRF «тихе злиття» (TC-11, 2.0.3-TEST-2)

FTS=1.0, Vector=1.0, **Hybrid=0.0**. RRF-функція silently падала при порожньому одному зі списків → повертала `[]`. **Виправлено** у 2.1.x (захист від порожніх списків).

### FP-2 — FTS latency spikes (46 s)

Початкова FTS5-індексація без `external-content` таблиць давала спайки 46 s. Оптимізовано індексними тригерами у 2.1.x.

### FP-3 — Dense сліпий до точних назв (GPG, Pydantic R@5=0.00)

Dense-retrieval (MiniLM) систематично не знаходить точні ідентифікатори (GPG key ID, Pydantic `model_dump`). Фундаментальна слабкість dense на термінальному пошуку.

### FP-4 — Крос-лінгвальна деградація MiniLM (UA→EN, MAR@5=0.208 у TEST-3)

`paraphrase-multilingual-MiniLM-L12-v2` дає помітний спад при mismatch мов запиту та документа (UA-запит → EN-документ). Це пояснює низький MRR_S у бенчмарку 2.2.2 (див. розділ 3).

### FP-5 — Reranker погіршує результати (MAR@5 0.423→0.329 у TEST-4, ×8 повільніше)

Cross-encoder ререйк на обрізаному тексті (короткий `RERANK_TEXT_CHARS`) та на розмитому RRF-pool (через слабкий vector) дає **гірший** MAR за FTS. Підтверджено у 2.2.2 (MRR 0.65→0.28).

### FP-6 — Методологічна нестабільність ground-truth (суперечність звітів)

- `2.0.3-TEST-2`: "Vector — переможець" (MAR@5 vector=0.65).
- `2.2.1`: FTS MRR=0.889 > hybrid 0.792 > vector 0.392.

**Корінь**: різні GT (ручний 3-рівневий labelling vs детермінований keyword-presence). Різні GT → протилежні висновки. Це головна причина, чому звіти суперечили одне одному. **Виправлено у 2.2.2** впровадженням **дуального GT** (розділ 3).

### FP-7 — Тихі відмови (Silent failures)

2.2.1 BUG-01: string-path → 0 результатів без помилки. Також FP-1 (Hybrid=0.0) та FP-5 (rerank гірший) маскуються відсутністю явних помилок. Висновок: **TIХІ відмови небезпечніші за краші** — вони проходять CI та виглядають як "погана якість", а не баг.

## 🧪 3. IR-бенчмарк v2.2.2: повний vault, дуальний ground-truth

### Методологія (виправлена)

- **Корпус**: реальний vault `/root/gemma/brain` (**565 .md**, змішаний UA/EN). **Усі 5 режимів виміряно на повному vault** (на відміну від 2.2.1, де semantic/hybrid_reranked обмежувалися під-набором 100 файлів).
- **Embeddings**: `paraphrase-multilingual-MiniLM-L12-v2`, `EMBED_NUM_THREADS=2`. Live DB: `/tmp/power_minilm_565.db` (507 файлів проіндексовано, 808 chunk_embeddings).
- **Дуальний ground-truth** (усуває FP-6):
    - **GT-LEXICAL** (детермінований): нотатка релевантна, якщо УСІ ключові слова (len>2) присутні у `rel_path` або тілі. FTS-дружній.
    - **GT-SEMANTIC** (ручне курування): 20 запитів з явними шляхами до реальних нотаток vault. Dense-дружній.
- **Запити**: 20 (дзеркальні до 2.0.3-TEST-2 + крос-лінгвальні UA).
- **Метрики**: MRR, nDCG@10, Recall@10, latency (warm model) — окремо для L (лексичний GT) та S (семантичний GT).
- **Обладнання**: WS, 126 ГБ RAM, 20 cores.

### Результат — повний vault (565 файлів), УСІ 5 режимів

| Режим                        |   MRR_L   |  nDCG_L   |   Rec_L   |   MRR_S   |  nDCG_S   |   Rec_S   |  Latency  |
| :--------------------------- | :-------: | :-------: | :-------: | :-------: | :-------: | :-------: | :-------: |
| **fts** (BM25/FTS5)          | **0.650** | **0.647** | **0.366** |   0.029   |   0.045   |   0.050   | **17 ms** |
| vector (TF cosine)           |   0.192   |   0.168   |   0.043   |   0.081   |   0.124   |   0.175   |  405 ms   |
| hybrid (RRF FTS+Vector)      |   0.592   |   0.581   |   0.342   |   0.110   |   0.146   |   0.175   |  433 ms   |
| semantic (dense MiniLM)      |   0.149   |   0.165   |   0.070   | **0.298** | **0.313** | **0.425** | 21358 ms  |
| hybrid_reranked (RRF+Rerank) |   0.276   |   0.321   |   0.271   |   0.062   |   0.070   |   0.050   | 32846 ms  |

Агрегат збережено: `/tmp/ir_bench_22_minilm.json`.

### Ключові висновки (остаточні, дуальний GT)

1. **FTS (BM25) — найкращий за лексичним GT** (MRR_L 0.650, 17 ms). Домінує на keyword/ID-пошуку. Підтверджує FP-3: точні назви (GPG, Pydantic) знаходить лише FTS.
2. **Semantic (dense MiniLM) — найкращий за семантичним GT** (MRR_S 0.298, Rec_S 0.425), АЛЕ найповільніший (**21 s**). Підтверджує, що dense-retrieval ефективний лише для семантичних (не-лексичних) запитів.
3. **Hybrid (RRF FTS+Vector) — компроміс** (MRR_L 0.592 ≈ FTS; MRR_S 0.110 > FTS 0.029). Кращий за FTS на semantic-GT, але поступається FTS на lexical-GT (ефект FP-1/RRF «weakest link» — vector розмиває FTS).
4. **hybrid_reranked — найгірший за обома GT** (MRR_L 0.276, MRR_S 0.062) і найповільніший (**33 s**). Підтверджує FP-5: rerank погіршує результат через слабкий RRF-pool + обрізаний текст. **Не рекомендується до перебудови.**
5. **Крос-лінгвальна деградація (FP-4)**: низький MRR_S загалом (semantic 0.298 — це максимум) пояснюється UA↔EN mismatch у vault. MiniLM multilingual все ще слабкий на cross-lingual.
6. **Розв'язання суперечності FP-6**: дуальний GT показує, що жоден режим не "перемагає універсально". FTS кращий для lexical, semantic — для semantic. Вибір режиму має залежати від типу запиту, а не від "найкращого середнього".

## 🌐 4. Огляд Best Practices (липень 2026)

Актуальні джерела підтверджують емпіричні висновки:

1. **Hybrid (BM25 + dense) + RRF — production standard**, +15–30% recall над dense-only. Підтверджує: hybrid (MRR_L 0.592) > vector (0.192).
2. **Cross-encoder reranking** — головний драйвер точності, АЛЕ лише при якісному candidate-pool. Наш hybrid_reranked програє через слабкий pool (FP-5).
3. **BM25 часто кращий за dense на точних термінах/ID** (FP-3). Пояснює домінування FTS (MRR_L 0.650).
4. **Chunking — найвпливовіша змінна** (>±25% accuracy swing). Наші 808 chunks/565 файлів — достатньо, але `SemanticChunker` дає 0 chunks для коротких нотаток → розріджене покриття.
5. **RRF «weakest link»** (FP-1): слабкий retrieval-path розмиває сильний. Пояснює чому hybrid (0.592) < FTS (0.650) на lexical-GT.
6. **UDCG** — нова RAG-орієнтована метрика (utility + distraction-aware). Рекомендується для майбутніх бенчмарків замість чистого nDCG.
7. **Contextual/hierarchical chunking** покращує retrieval precision.

### Рекомендації (Roadmap)

- ✅ За замовчуванням використовувати **`fts`** для lexical-запитів, **`semantic`** для semantic-запитів (determine via query-classifier).
- 🔧 Перебудувати `hybrid_reranked`: ререйкати FTS-кандидатів (top-K за BM25) cross-encoder-ом, збільшити `RERANK_TEXT_CHARS`.
- 🔧 Додати graceful fallback: якщо chunk_embeddings порожні → semantic повертає FTS-результати (зараз повертає `[]`).
- 🔧 `SemanticChunker` має повертати whole-doc chunk для коротких нотаток (уникнути 0-chunk).
- ⚡ Розглянути легшу cross-lingual модель (bge-m3) замість MiniLM для покращення MRR_S (FP-4).
- 🛡️ Додати RSS-limit/warning у `power sync` (інцидент 42 ГБ).
